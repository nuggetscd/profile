## 一、项目概述

### 1.1 背景与动机

本项目源于一个真实的个人痛点：在学习法语的过程中，刷抖音、B站等中文短视频平台会带来明显的时间浪费感，但完全停止娱乐也不现实。现有方案（字幕翻译插件、截图翻译 App）均要求用户主动注意屏幕文字，打断了视觉体验，无法实现真正的被动语言输入。

**核心理念**：将"刷视频"变成"被动语言沉浸"——视频照常播放，耳机里同步听到目标语言的翻译，用户无需分散注意力即可持续获得语言输入。

### 1.2 产品定位

| 维度 | 描述 |
|------|------|
| 产品类型 | macOS 桌面端命令行工具（CLI） |
| 核心功能 | 实时捕获系统音频 → 中文语音识别 → 目标语言翻译 → TTS 语音播放 |
| 目标用户 | 正在学习外语、日常刷中文短视频的用户 |
| 使用场景 | 刷抖音/B站/微博视频时，耳机里同步听到法语/西班牙语翻译 |
| 技术形态 | Python + 本地 Whisper 模型 + DeepSeek API + edge-tts |

## 二、用户研究与需求分析

### 2.1 目标用户画像

| 维度 | 描述 |
|------|------|
| 用户群体 | 在学外语（法/西/日/韩语）的中国用户，日常重度使用短视频平台 |
| 核心痛点 | 刷视频感觉浪费时间，但学语言又需要大量时间输入，两者冲突 |
| 现有行为 | 偶尔开字幕翻译插件，但需要盯屏幕看字幕，影响视频观看体验 |
| 期望状态 | 娱乐和语言学习同步发生，不需要额外付出注意力 |

### 2.2 需求拆解

**核心需求（Must Have）**
- 实时识别视频中的中文语音
- 将识别结果翻译成目标语言（法语优先）
- 通过 TTS 将译文朗读至耳机
- 不影响视频原声的正常播放

**重要需求（Should Have）**
- 检测说话停顿，按自然句子边界切割，而非固定时间窗口
- TTS 播放期间暂停音频捕获，防止自我反馈死循环
- 静音片段自动跳过，不触发识别

**待优化（Nice to Have）**
- 延迟从 3-5 秒优化至 1-2 秒（接入流式 ASR）
- 支持 Windows 平台
- GUI 控制界面（启停、切换语言）

## 三、技术方案与选型决策

### 3.1 系统架构

```
系统音频（BlackHole）→ Whisper 语音识别 → DeepSeek 翻译 → edge-tts → AirPods
```

四个模块独立运行于不同线程/异步循环，互不阻塞。

### 3.2 技术选型对比

| 模块 | 选用方案 | 备选方案 | 选型理由 |
|------|----------|----------|----------|
| 音频捕获 | BlackHole + sounddevice | Loopback（付费） | 唯一免费的 macOS 系统音频捕获方案，无需 root 权限 |
| 语音识别 | faster-whisper（本地） | Google Cloud STT | 零 API 成本，离线可用，中文识别准确率高；Google STT 延迟更低但 60 分钟/月免费额度不够日常使用 |
| 翻译 | DeepSeek API | GPT-4o / DeepL | 中文→法语翻译质量与 GPT-4o 相当，成本低约 10 倍；DeepL 不支持中文 |
| 语音合成 | Microsoft edge-tts | OpenAI TTS | 完全免费，Neural 语音自然度高，支持 100+ 语种；OpenAI TTS 按字符计费 |
| 音频切割 | VAD（语音活动检测） | 固定时间窗口 | 按自然停顿切割，延迟比固定 5 秒窗口降低约 40%，翻译颗粒度更贴近语义单元 |

### 3.3 关键技术决策说明

**为什么选择语音输出而非字幕显示？**

字幕方案要求用户主动阅读屏幕，与视频内容争夺视觉注意力，无法实现被动输入。语音输出通过耳机传递，与视频音轨分离，用户无需额外操作即可持续获得语言输入，符合沉浸式语言学习的理论基础（可理解性输入，Krashen 1982）。

**VAD 切割 vs 固定时间窗口**

固定 5 秒窗口存在两个问题：①即使无人说话也会触发识别，浪费计算资源；②一句话可能被切割在两个窗口中，导致语义不完整。VAD 通过实时检测音量 RMS 值判断说话/停顿，在自然停顿处切割，平均段长 2-3 秒，端到端延迟从 8 秒降至 3-5 秒。

**防反馈机制设计**

TTS 朗读的法语音频同样会被 BlackHole 捕获，若不处理会触发"识别法语→翻译成更多法语"的死循环。通过全局标志位 `is_speaking`，在 TTS 播放期间暂停音频队列入队，播放完毕后恢复，彻底消除反馈问题。

## 四、产品指标与体验评估

### 4.1 核心指标

| 指标 | 当前值 | 目标值 | 瓶颈说明 |
|------|--------|--------|----------|
| 端到端延迟 | 3-5 秒 | 1-2 秒 | 本地 Whisper 为批处理模式，需等待完整音频片段 |
| 识别准确率 | 普通话约 90% | >90% | 方言及背景噪音场景下下降明显 |
| 翻译质量 | 自然流畅 | 保持 | DeepSeek 对口语化表达处理良好 |
| TTS 自然度 | 中等（AI 感明显） | 待优化 | Neural TTS 语调较平，无情绪起伏 |
| API 成本 | 约 ¥0.002/句 | <¥0.01/句 | DeepSeek 定价极低，日常使用成本可控 |

### 4.2 用户体验评估（自测结论）

- ✅ **核心体验达标**：刷视频时可在不看屏幕的情况下持续听到法语翻译，基本实现被动输入目标
- ✅ **延迟基本可接受**：3-5 秒延迟在视频场景下可接受，用户已听完中文原声，法语作为回响
- ✅ **静音跳过有效**：背景音乐/无人说话片段不触发识别，节省约 60% 无效计算
- ⚠️ **TTS 自然度有限**：Microsoft Denise Neural 已是免费方案最优，但与真人播音仍有明显差距

## 五、已知局限与迭代规划

### 5.1 当前局限

| 问题 | 影响程度 | 根本原因 |
|------|----------|----------|
| 延迟 3-5 秒 | 高（对话类视频体验较差） | 本地 Whisper 为批处理模式，非流式识别 |
| 仅支持 macOS | 高（Windows 用户无法使用） | 依赖 BlackHole 虚拟声卡和 afplay 播放器 |
| 仅支持中文输入 | 中（英文/日语视频无法翻译） | Whisper 固定 language 参数为 zh |
| TTS 自然度有限 | 中（长时间收听有疲劳感） | 免费 TTS 方案的固有局限 |

### 5.2 迭代路径

| 优先级 | 方向 | 方案 | 预期收益 |
|--------|------|------|----------|
| P0 | 降低延迟 | 接入 Google Cloud 流式 STT 或 Whisper streaming 模式 | 延迟从 3-5 秒降至 1-2 秒 |
| P1 | 多语言输入 | 去除 language 固定参数，改为 Whisper 自动语言检测 | 支持英/日/韩语视频 |
| P2 | Windows 支持 | 用 VB-Cable 替代 BlackHole，用 pygame 替代 afplay | 扩大用户覆盖范围 |
| P3 | GUI 界面 | 用 tkinter 或 PyQt 做简单控制面板 | 降低使用门槛至非开发者 |

## 六、总结

本项目从真实个人需求出发，在技术选型上优先考虑成本与实用性的平衡：本地 Whisper 保证零边际成本，DeepSeek API 提供高质量低成本翻译，edge-tts 实现免费自然语音合成。VAD 切割和防反馈机制解决了实际使用中遇到的工程问题。

当前产品已完成 MVP 验证，核心功能稳定运行。主要瓶颈在于延迟（3-5 秒）和 TTS 自然度，均有明确的优化路径。下一步将优先接入流式 ASR 以改善实时体验。
